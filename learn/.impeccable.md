## Design Context

### Users
Developer preparing for an Apple Instruments team internship. Using this app to rapidly learn profiling fundamentals, OS internals, and platform-specific tooling (Chrome, Android, iOS). Context is solo study — high motivation, short timeline, needs to absorb dense technical material fast without wading through walls of text. Already comfortable with web dev and Chrome DevTools, new to systems-level concepts.

### Brand Personality
Precise. Focused. Engineered.

The interface should feel like a well-made instrument itself — every element earns its place, nothing decorative, nothing wasted. Think of the confidence you feel opening a tool built by someone who clearly knows what they're doing.

### Aesthetic Direction
- **Visual tone**: Technical, clean, high-contrast dark. The density of an engineering dashboard without the clutter. Closer to a terminal or IDE than a classroom.
- **References**: Linear (information density done right, surgical use of color, no unnecessary chrome), Exercism (developer-focused learning, structured progression, clean cards)
- **Anti-references**: Duolingo (no mascots, no confetti, no streaks-as-guilt, no infantilizing). No Udemy course pages. No dense docs sites. No AI-generated gradient blobs.
- **Theme**: Dark mode only. This is a tool for night-owl developers, not a marketing site.
- **Color**: Purple primary already set (`hsl(262 83% 58%)`). Use it surgically — active states, progress indicators, selected items. Everything else is neutral grayscale. Green for correct/success, amber for XP, red for wrong. That's it.

### Design Principles

1. **Density over decoration** — Show more information in less space. No hero sections, no padding-heavy layouts. Every pixel teaches something. If a section could be a single well-written sentence, don't make it a paragraph.

2. **Structure is the UI** — The lesson hierarchy (OS > Instruments > Chrome > Android > iOS) IS the navigation. Progress bars, module cards, and breadcrumbs replace prose explanations of "what you'll learn." Let the structure speak.

3. **Code and diagrams are first-class** — Mermaid diagrams, ASCII art, and code blocks should feel native, not embedded. Monospace text is as important as prose. Diagrams should have the same visual weight as text — not decorative illustrations but actual teaching tools.

4. **Earned complexity** — Each lesson builds on the last. The UI should reflect this: locked/unlocked states, prerequisite indicators, and a clear sense of "you are here." Not gamified — engineered. Progress is a data point, not a reward.

5. **Immediate feedback** — Quizzes answer on click, not on submit. Correct = brief green flash + explanation. Wrong = the explanation IS the learning moment. No "try again" loops — show the answer, explain why, move on. Respect the user's time.

### Typography
- **Headings**: Geist Variable (already loaded), 600 weight, tight tracking
- **Body**: Geist Variable, 400 weight, 16px base, 1.6 line-height for readability in technical content
- **Code / diagrams**: JetBrains Mono (defined in theme), same visual weight as body text
- **Scale**: 14 / 16 / 20 / 24 / 32 — five sizes, no more

### Spacing
- Base unit: 4px
- Component padding: 12px / 16px / 24px
- Section gaps: 32px / 48px
- Max content width: 720px for lesson content (readability), full-width for diagrams and timelines

### Component Patterns
- **Lesson cards**: Border-left accent color indicating module, minimal interior — title, subtitle, status badge
- **Quiz questions**: Radio groups with immediate reveal. Selected-wrong gets a muted red left-border. Selected-correct gets green. Explanation appears inline below.
- **Mermaid diagrams**: Dark theme matching app background, purple for highlighted nodes, neutral for structure
- **Progress**: Thin horizontal bars, not circular. Numbers over animations.
- **Navigation**: Left sidebar with module tree, collapsed on mobile. Current lesson highlighted. No breadcrumbs needed if sidebar is always visible.
